\section{Обоснование требований задания на дипломное проектирование}

Разрабатываемое программное средство представляет собой сложную вычислительную систему, реализующую асимметричный гибридный алгоритм сжатия и нейросетевого восстановления цифровых изображений. Процесс разработки, оценка качества и формирование эксплуатационной документации должны осуществляться в строгом соответствии с действующими государственными стандартами Республики Беларусь (СТБ) и межгосударственными стандартами (ГОСТ).

Оценка качества программного обеспечения должна производиться с учетом номенклатуры показателей, регламентированных ГОСТ 28195-89 «Оценка качества программных средств. Общие положения» \cite{gost_28195_89}. В соответствии с данным стандартом, к проектируемой системе предъявляются требования по надежности, эффективности, эргономичности и переносимости. Жизненный цикл разрабатываемого программного продукта должен соответствовать методологии, описанной в СТБ ИСО/МЭК 12207-2003 \cite{stb_iso_12207}.

\subsection{Требования к функциональным характеристикам}

Функциональное назначение разрабатываемой системы заключается в обеспечении процесса компрессии растровых графических данных на стороне отправителя с минимальными затратами энергии и их высококачественной реконструкции на стороне получателя (модель Edge-to-Cloud).

В соответствии с ГОСТ 19.201-78 \cite{gost_19_201_78}, программный комплекс должен выполнять следующие основные функции:

\begin{enumerate}
    \item Загрузка исходных растровых изображений (в том числе пакетная) пользователем через графический веб-интерфейс.
    \item Предварительная оценка информационной емкости файла модулем адаптивной маршрутизации (Smart Router) для принятия решения о целесообразности сжатия (с порогом отсечения в 50 Кбайт).
    \item Динамическая экстракция гибридных метаданных: формирование цветового паспорта малого разрешения и бинарной карты физических границ объектов (алгоритм Кенни).
    \item Осуществление каскадного дискретного вейвлет-преобразования (базис Хаара) с автоматическим или ручным выбором глубины декомпозиции (уровни 1--3) для обеспечения коэффициента пространственного сжатия от 4:1 до 64:1.
    \item Упаковка полученного вейвлет-скелета в стандартный формат JPEG (энтропийное кодирование) для симуляции передачи пакета по узкополосному каналу связи.
    \item Декодирование полученного пакета и выполнение интеллектуального апскейлинга с использованием интегрированных моделей сверточных сетей (ESPCN или FSRCNN).
    \item Семантическая постобработка восстановленного тензора: рекомбинация переданных метаданных (цветокоррекция интерполяцией Ланцоша и локальное повышение микроконтраста по маске).
    \item Отображение этапов обработки, расчет и вывод объективных метрик качества (PSNR, SSIM), коэффициента сжатия трафика, а также раздельного времени работы кодера и декодера.
\end{enumerate}

\subsection{Требования к надежности}

Надежность программного обеспечения, согласно ГОСТ 28195-89 \cite{gost_28195_89}, определяется его способностью сохранять работоспособность при установленных условиях использования. Для обеспечения надежности разрабатываемая система должна удовлетворять следующим критериям:

\begin{enumerate}
    \item Защита от исчерпания памяти (Out-Of-Memory): поскольку обработка изображений сверхвысокого разрешения (4K и выше) с применением нейросетей требует значительных объемов оперативной памяти, система обязана реализовывать алгоритм тайловой маршрутизации с нахлестом (Overlapping Tiling). Изображение должно динамически разрезаться на перекрывающиеся блоки для обеспечения строго линейного потребления RAM.
    \item Устойчивость к отказам: программа должна корректно обрабатывать исключительные ситуации (например, поврежденные графические файлы). При возникновении ошибки система не должна аварийно завершать работу сервера; проблемный файл должен исключаться из пакета обработки.
    \item Механизм кэширования: для проведения многократных лабораторных испытаний система должна кэшировать пути к загруженным файлам, чтобы избавить пользователя от необходимости повторной передачи данных на сервер при смене гиперпараметров кодека.
\end{enumerate}

\subsection{Требования к программной и аппаратной совместимости}

Разрабатываемая архитектура базируется на асимметричной модели вычислений, поэтому к аппаратному обеспечению различных узлов предъявляются раздельные требования:

\begin{itemize}
    \item Узел-отправитель (Клиент): устройство пользователя (в том числе мобильное), к которому предъявляются минимальные требования. Достаточно наличия современного веб-браузера и любого процессора архитектуры ARM или x86 для выполнения легковесных математических операций DWT-кодирования.
    \item Узел-получатель (Сервер): многоядерный центральный процессор (CPU) архитектуры x86-64 с поддержкой векторных инструкций AVX2. Объем оперативной памяти (RAM) — не менее 8 ГБ. Использование энергоэффективных сетей ESPCN/FSRCNN, интегрированных через бэкенд OpenCV DNN, позволяет выполнять декодирование в приемлемые сроки исключительно силами CPU, без строгой необходимости в дорогостоящих графических ускорителях (GPU).
\end{itemize}

Программная часть (серверная логика) должна функционировать под управлением операционных систем семейства GNU/Linux или Microsoft Windows. В качестве среды выполнения должен использоваться интерпретатор Python версии 3.8 или выше. Реализация веб-сервера должна базироваться на микрофреймворке Flask.

\subsection{Требования к эргономике и технической эстетике}

Взаимодействие пользователя с системой должно быть организовано в соответствии с принципами человеко-ориентированного проектирования интерактивных систем (ГОСТ Р ИСО 9241-210-2012 \cite{gost_iso_9241_210}). Графический интерфейс должен минимизировать когнитивную нагрузку на оператора.

Разработанный графический пользовательский интерфейс (GUI) должен:
\begin{enumerate}
    \item Быть выполнен в адаптивной темной цветовой палитре (Dark Mode) для снижения зрительного утомления при длительной лабораторной работе с визуальными данными.
    \item Предоставлять два режима взаимодействия: переключатель между «Автоматическим режимом» (Smart Auto), скрывающим технические детали, и «Лабораторным стендом» (Manual) для ручной настройки глубины DWT и метаданных.
    \item Отображать результаты в виде наглядной горизонтальной ленты (сетки), последовательно демонстрирующей этапы: «Оригинал», «Пакет передачи» и «Восстановленный итог».
    \item Обеспечивать возможность детального инспектирования пикселей с помощью встроенного механизма увеличения изображений (Lightbox) по клику.
\end{enumerate}

\subsection{Требования к программной документации}

Состав и оформление программной документации должны строго соответствовать Единой системе программной документации (ЕСПД). Согласно ГОСТ 19.101-2000 \cite{gost_19_101_2000}, для разработанного программного комплекса должны быть подготовлены:
\begin{enumerate}
    \item Текст программы (исходный код на языке Python с необходимыми комментариями).
    \item Описание программы (сведения о логической структуре, функционировании алгоритмов DWT, экстракции метаданных и нейросетевого синтеза).
    \item Руководство оператора (инструкция по развертыванию серверного окружения через менеджер пакетов \texttt{pip} и эксплуатации графического интерфейса).
\end{enumerate}
Выполнение описанных требований гарантирует создание качественного, отказоустойчивого программного прототипа, готового к проведению экспериментальных исследований.
